# Phase 2.5B: Beauty Log リデザイン — メイク × AI × コスメデータ

## Context

ユーザーから「既存の Journal アプリから乗り換えたいと思えるレベルに」との要望。

**競合分析**: Day One（自動メタデータ、Streaks）、Daylio（Year in Pixels、Stats）、Apple Journal（写真ファースト）— いずれも**受動的な記録ツール**。

**alche:me の差別化**: 記録するだけでなく、AI が学習して**能動的に提案する**ビューティーダイアリー。

### 差別化の核（他のジャーナルアプリにないもの）

| 機能 | Day One | Daylio | alche:me |
|------|---------|--------|----------|
| 天気自動記録 | ✅ | ❌ | ✅ + 気温・湿度・美容アドバイス |
| 統計/パターン | ❌ | ✅ (Stats) | ✅ + **AI 洞察**（天気×評価相関、コスメ別パフォーマンス） |
| Today's Forecast | ❌ | ❌ | ✅ **天気+過去パターン → 今日のおすすめレシピ** |
| 製品トラッキング | ❌ | ❌ | ✅ **used_items → どのコスメが高評価に貢献？** |
| レシピ連携 | ❌ | ❌ | ✅ レシピ ↔ ログ双方向リンク |
| ストーリーズ | ❌ | ❌ | ✅ 過去7日のルック + 今日の提案カード |
| ヒートマップ暦 | ❌ | ✅ (Pixels) | ✅ Beauty Palette + 写真サムネイル |

### 既存コードベースの活用

重要な発見: 以下が**既に実装済み**だが UI で未活用:
- `used_items: string[]` — スキーマにあるが UI で空配列のまま
- `auto_tags: string[]` — スキーマにあるが未使用
- `agent/alcheme/tools/profiler_tools.py` — 30ログ+20レシピの傾向分析
- `agent/alcheme/tools/weather_tools.py` — 天気+湿度+UV+美容アドバイス（agent側は既に humidity 対応済み）
- `agent/alcheme/tools/context_tools.py` — 季節キーワード、眠っているコスメ、最近のレシピ

## Batch 1: データ基盤（Schema + API + used_items 自動埋め）

### 1A. BeautyLogEntry スキーマ拡張

**`types/beauty-log.ts`** — 2フィールド追加:
```typescript
temp?: number;      // 気温 °C (自動取得)
humidity?: number;   // 湿度 % (自動取得)
```

### 1B. Weather API 湿度返却（フロントエンド用）

**`app/api/weather/route.ts`**:
- `fetchOpenMeteo()`: `current` パラメータに `relative_humidity_2m` 追加
- レスポンス: `{ weather, temp, humidity, source }`
- 注: agent 側 (`weather_tools.py`) は既に humidity 対応済み。フロントエンド API のみ修正

### 1C. Beauty Log API 拡張

**`app/api/beauty-log/route.ts`**:
- POST: `body.temp`, `body.humidity` のハンドリング追加
- POST: `recipe_id` がある場合、レシピの steps から `used_items` を自動埋め（レシピ doc を取得 → steps[].item_id を抽出）
- GET: レスポンスに `temp`, `humidity` を含める
- GET: `start`/`end` 日付範囲クエリパラメータ追加（Forecast/Report 用）

**`app/api/beauty-log/[logId]/route.ts`**:
- GET: `temp`, `humidity` を返却に追加

### 1D. フォーム メタデータ強化

**`components/beauty-log-form.tsx`** — 全フィールド表示のまま:
- 天気自動取得で `temp`, `humidity` も取得・state に保持
- 天気セクションにリッチ表示: 「☀️ 晴れ 12°C 湿度45%」
- フォーム送信時に `temp`, `humidity` を POST body に含める

4. 美容アドバイスは天気条件から生成（`weather_tools.py` の `_WEATHER_BEAUTY_TIPS` と同等のマッピングをフロントに持つ）

## Batch 3: Weekly Beauty Report + AI 洞察（差別化の核）

### 3A. 集計ロジック

**NEW `lib/beauty-log-analytics.ts`** — 純粋関数:
```typescript
interface WeeklyReport {
  // 基本統計
  daysRecorded: number;
  averageRating: number | null;
  currentStreak: number;
  bestStreak: number;

  // Best Look（最高評価ログ）
  topRatedLog: BeautyLogEntry | null;

  // パターン分析（Profiler と同等のクライアント版）
  moodDistribution: Array<{ mood: string; count: number; avgRating: number }>;
  weatherCorrelation: Array<{ weather: string; avgRating: number; count: number }>;
  occasionDistribution: Array<{ occasion: string; count: number; avgRating: number }>;

  // 環境データ
  avgTemp: number | null;
  avgHumidity: number | null;

  // AI 洞察（ルールベース）
  insights: string[];
}

function computeWeeklyReport(logs: BeautyLogEntry[]): WeeklyReport;
```

**`insights` 生成ロジック**（ルールベース、agent 呼び出し不要）:
- 天気×評価: 「雨の日の平均★4.5 > 晴れの日の★3.8 → 雨メイクが得意！」
- 気分×評価: 「ウキウキの日は平均★4.7 — 気分がメイクに反映されていますね」
- シーン偏り: 「今週は仕事メイクが5/7日。週末は違うスタイルに挑戦しては？」
- Streak: 「5日連続記録！素晴らしいですね」
- Top Look: 「2/15のデートメイクが今週のベストルック ★5」

## 設計判断

1. **フォームは折りたたまない** — ユーザーフィードバックに基づき全フィールド常時表示
2. **Today's Forecast = クライアントサイド** — `/api/weather` + 過去ログから計算。agent 呼び出し不要（高速、オフライン対応）。より詳しい分析が欲しい場合はチャットで agent に聞ける導線を用意
3. **used_items 自動埋め = API サーバーサイド** — POST 時に recipe_id があればレシピ doc から steps[].item_id を抽出。UI 変更不要で既存データが充実
4. **AI 洞察 = ルールベース** — Profiler agent の `analyze_preference_history` と同等のパターン検出をフロントで実装。7件のログなのでクライアント計算で十分。将来的に agent 呼び出しに拡張可能
5. **天気美容アドバイス = フロント移植** — agent 側 `weather_tools.py` の `_WEATHER_BEAUTY_TIPS` + `_humidity_advice` をフロントにも持つ（Today's Forecast 用）
6. **Stories Tray + Forecast = スクロール内配置** — z-index 競合回避、Instagram 同様のスクロールアウト挙動
7. **日付範囲クエリ**（start/end）— 既存の Firestore date フィールドインデックスをそのまま利用
