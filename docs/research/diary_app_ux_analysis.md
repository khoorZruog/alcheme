## 1. エグゼクティブサマリー

### 調査の結論

ほぼ日手帳と Day One は、それぞれ「手書きの温かみ × コレクション感」と「テクノロジー × 自動化」というまったく異なるアプローチで、ユーザーの日記習慣を定着させることに成功している。alche:me の Beauty Log は、この**両者のベストプラクティスを「美容」という文脈に翻訳する**ことで、競合不在のポジションを獲得できる。

### 3つの核心インサイト

| # | インサイト | 出典 | Beauty Log への示唆 |
|---|-----------|------|-------------------|
| 1 | **「1日1ページ」の制約が自由を生む** | ほぼ日手帳 | 入力項目を絞り、「今日の顔」1枚 + 5段階評価だけで記録完了できる最小フローが必要 |
| 2 | **自動メタデータが記録の負担を激減させる** | Day One | 天気・位置情報・時刻を自動取得し、ユーザーの入力コストをゼロに近づける |
| 3 | **「振り返り」こそが継続のエンジン** | 両者共通 | 週次/月次の自動サマリーが「もっと記録したい」という欲求を生む好循環を作る |

## 2. ほぼ日手帳（Hobonichi Techo）分析

### 2.1 プロダクト概要

ほぼ日手帳は糸井重里氏率いる「ほぼ日」が2002年から展開する紙の手帳ブランド。物理プロダクトとしての完成度が極めて高く、年間約90万部を販売。デジタルコンパニオンとして「ほぼ日の學校」アプリや、手帳の使い方を共有する「ほぼ日手帳 みんなの使い方」Web コンテンツが存在する。

### 2.2 コア UX デザイン原則

#### A. 「1日1ページ」の設計哲学

| 原則 | 詳細 |
|------|------|
| **ちょうどいい制約** | A6サイズ・1日1ページという物理的制約が「何を書くか」の意思決定を軽くする。ページが大きすぎると空白が怖くなり、小さすぎると表現が窮屈になる。この「ちょうどいいサイズ」の概念はデジタルUIにも応用可能 |
| **書かなくてもいい自由** | 公式が「書かない日があっても大丈夫」と明言。罪悪感を生まない設計がむしろ継続率を上げている |
| **方眼ノート** | 3.7mm方眼は文字にも絵にも対応。構造を押し付けず、ユーザーの自由な表現を許容する |
| **日付だけが印刷済み** | テンプレートは最小限（日付・曜日・月の満ち欠け・日々の言葉）。残りはすべて白紙。この「余白のデザイン」が重要 |

#### B. 「LIFEのBOOK」コンセプト

ほぼ日手帳の本質は「スケジュール管理ツール」ではなく「人生を記録する本」。この哲学転換が以下を実現している：

- **記録することの意味づけ**: 予定を書くのではなく、「今日あったこと」を書く。過去を振り返る喜びが動機になる
- **コレクションとしての価値**: 1年で1冊が完成する。本棚に並ぶ手帳は「自分の人生の蔵書」になる
- **完璧主義からの解放**: 公式が「落書きでもシールでもレシートを貼るだけでも」と推奨。ハードルを徹底的に下げている

#### C. ナビゲーションパターン（物理プロダクト）

| パターン | 説明 | デジタルへの翻訳 |
|---------|------|-----------------|
| **しおり（2本）** | 今日のページと、よく参照するページにアクセス。物理的ブックマーク | 「今日」ボタン + お気に入りログへのクイックアクセス |
| **月間カレンダー → 日別ページ** | 月間俯瞰から日別へドリルダウン | カレンダービュー → 日別詳細の遷移。alche:me の現在の実装と同じ |
| **年間インデックス** | 年間を通した俯瞰。月ごとの目標やイベントを一望 | 年間ヒートマップ/統計ダッシュボード |
| **巻末メモページ** | 自由に使えるまとまったスペース | フリーテキストノート機能 |

### 2.3 日々の記入フロー

```
手帳を開く（物理的な「儀式」）
    ↓
日付は印刷済み（ゼロ入力）
    ↓
「日々の言葉」を読む（小さな発見・モチベーション）
    ↓
自由に記入（文字・絵・写真・シール、何でもOK）
    ↓
閉じる（達成感）
```

**ポイント**: 「開く」という物理的アクションが儀式化している。デジタルでは「アプリを開いた瞬間に今日の入力画面が表示される」ことがこれに相当する。

### 2.4 テンプレート/プリセットシステム

ほぼ日手帳は意図的にテンプレートを最小限にしている：

- **日付・曜日**: 自動印刷
- **月の満ち欠け**: 自動印刷
- **日々の言葉（日本語＋英語）**: 毎日異なる著名人の言葉を掲載。これが「今日手帳を開く理由」の一つになっている
- **ToDoリスト枠（上部）**: 小さなチェックボックス欄があるが使わなくても自由
- **時間軸（左端）**: 控えめな時間目盛り。使っても使わなくてもいい

**Beauty Log への示唆**: 「今日のビューティーTips」や「今日の美容成分豆知識」のような日替わりコンテンツを配置し、アプリを開く動機を作る。

### 2.5 コミュニティとソーシャル要素

ほぼ日手帳の成功は物理的な手帳だけでなく、それを囲むコミュニティが支えている：

| 要素 | 説明 |
|------|------|
| **「みんなの使い方」** | 公式サイトでユーザーの手帳の使い方を紹介。「こういう使い方もあるんだ」という発見が新しい使い方を触発する |
| **#ほぼ日手帳 ハッシュタグ** | Instagram/X 上で膨大な投稿数。他のユーザーの美しいページを見ること自体が楽しい |
| **ほぼ日手帳ミーティング** | オフラインイベント。手帳を見せ合い、使い方を共有する「文化祭」のような場 |
| **カバーの着せ替え文化** | 100種類以上のカバーから選ぶ楽しさ。手帳を「自分だけのもの」にするカスタマイズ欲求を満たす |

### 2.6 ビジュアルデザイン言語

| 要素 | 詳細 |
|------|------|
| **色彩** | 本体は白・クリーム色を基調とし、ユーザーの書き込みが映えるキャンバスに徹する。ブランドカラーは赤（ロゴ・帯） |
| **タイポグラフィ** | 印刷文字は控えめなサンセリフ。「主役はユーザーの手書き」という思想 |
| **余白** | 方眼は存在するが押し付けがましくない薄いグレー。余白を「デザインされた自由」として扱う |
| **紙質** | トモエリバー（薄くて丈夫、裏抜けしにくい）。触感・書き心地にこだわる＝デジタルでの「操作感」に相当 |

### 2.7 リテンション（継続率）の仕組み

| フック | 説明 | 有効性 |
|--------|------|--------|
| **日替わりコンテンツ** | 「日々の言葉」は毎日異なる。今日の言葉を読みたい、という小さな動機 | 高 |
| **サンクコスト** | 物理的に「書いたページ」が蓄積。捨てたくない＝続けたいという心理 | 非常に高 |
| **コレクション欲** | 年ごとに1冊。本棚に並ぶ手帳＝人生の記録のコレクション | 非常に高 |
| **カバーへの愛着** | お気に入りのカバーを選ぶ投資が、使い続ける動機に | 中 |
| **コミュニティ同調圧力** | みんなが使っている＝自分も続けたいというソーシャルプルーフ | 高 |
| **ゼロプレッシャー設計** | 書かない日があってもOK。罪悪感がないから「また書こう」と思える | 非常に高 |

### 2.8 振り返り機能

物理手帳としての振り返りは以下の形で設計されている：

- **月間カレンダーページ**: 月初に1ページ。ここに予定を書き、月末に振り返る
- **年間ページ**: 1年の計画と振り返り
- **1年後に読み返す**: 「去年の今日、何していたっけ？」という楽しみ。Day One の "On This Day" と同じ効果
- **My 100（100のリスト）**: 年始に100個のやりたいことを書く。年末にどれだけ達成したか振り返る

## 3. Day One Journal 分析

### 3.1 プロダクト概要

Day One は2011年にリリースされた iOS/macOS/Android/Web 対応のジャーナルアプリ。Apple Design Award を受賞し、プライベートジャーナリングアプリの代名詞的存在。2021年に Automattic（WordPress 親会社）に買収され、エンドツーエンド暗号化を搭載。

### 3.2 コア UX デザイン原則

#### A. 「ゼロフリクション記録」の設計思想

| 原則 | 実装 |
|------|------|
| **1タップで記録開始** | アプリアイコン長押し（3D Touch / Quick Actions）で即座に新規エントリー。ロック画面ウィジェットからも記録可能 |
| **メタデータの自動取得** | 天気・気温・位置情報・歩数・再生中の音楽・移動距離を自動記録。ユーザーの入力負担を極限まで削減 |
| **テンプレート先行** | 白紙からの記入だけでなく、「Gratitude（感謝）」「Daily Log」「Travel」等のテンプレートから始められる |
| **リマインダー** | 毎日決まった時間に通知。「今日はまだ書いていません」ではなく「今日を振り返ってみませんか？」という肯定的な言い回し |

#### B. 「プライベート・サンクチュアリ」としてのポジショニング

Day One はSNSとは真逆の「完全プライベート」を訴求：

- **エンドツーエンド暗号化**: 開発者すら読めない。この安心感が「本音を書ける」信頼を生む
- **SNS疲れの受け皿**: 「いいね」も「フォロワー」もない。自分だけの空間
- **書籍化機能**: ジャーナルを製本して物理的な本にできる。プライベートだが「形に残る」

### 3.3 ナビゲーションパターン

```
┌─────────────────────────────────────┐
│          Day One ナビゲーション構造      │
├─────────────────────────────────────┤
│                                     │
│  [Timeline]  [Calendar]  [Map]      │  ← 3つのビューモード
│      │           │         │        │
│      ▼           ▼         ▼        │
│  時系列      月間俯瞰    地図上     │
│  エントリー   ドットで     ピンで     │
│  フィード     記録日表示   記録地表示  │
│      │           │         │        │
│      └─────┬─────┘         │        │
│            ▼               │        │
│     エントリー詳細  ◄──────┘        │
│            │                        │
│            ▼                        │
│     編集 / 写真追加                  │
│                                     │
│  [+ 新規] ← FAB（Floating Action）  │
│                                     │
│  サイドバー:                         │
│    - 複数ジャーナル切替              │
│    - タグフィルター                   │
│    - On This Day                    │
│    - 検索                            │
└─────────────────────────────────────┘
```

#### 主要ナビゲーション要素

| 要素 | 説明 | Beauty Log への適用可能性 |
|------|------|-------------------------|
| **タイムラインビュー** | デフォルト表示。新しい順にエントリーが並ぶ。写真があれば大きくサムネイル表示 | 現在の alche:me タイムラインビューに相当。写真の見せ方を改善できる |
| **カレンダービュー** | 月間カレンダーにドット表示。記録がある日にはドット（色付き）が付く | 現在の alche:me カレンダービューに相当。ドットの表現を強化できる |
| **地図ビュー** | 地図上にエントリーのピンが立つ。旅行記録に最適 | Beauty Log では「イベント/場所別メイク」としてアレンジ可能 |
| **FAB（Floating Action Button）** | 画面右下に常駐する「+」ボタン。どの画面からでも即座に新規記録 | 現在の alche:me の「記録」ボタンをFABに変更し、常時アクセス可能にする |
| **サイドバー** | 複数ジャーナルの切替、タグ、検索等の高度な機能を格納 | BottomNav内で対応可能 |

### 3.4 日々の記入フロー

```
■ 最短フロー（10秒で完了）
   アプリ起動 → + タップ → 写真撮影 → 保存
   （天気・位置・時刻は自動記録済み）

■ 標準フロー（1-2分）
   アプリ起動 → + タップ → テンプレート選択 → テキスト入力
   → 写真追加 → ムード選択 → タグ付与 → 保存

■ リッチフロー（5分以上）
   アプリ起動 → + タップ → テンプレート選択 → テキスト入力
   → 複数写真 → 音声メモ → ムード → タグ → チェックリスト → 保存
```

**重要な設計ポイント**: 最短フローが10秒で完了すること。これが「毎日続けられる」の鍵。

### 3.5 テンプレート/プリセットシステム

Day One のテンプレートは非常に充実している：

| テンプレートカテゴリ | 例 | 構成要素 |
|--------------------|---|---------|
| **Daily Prompts** | 「今日感謝したことは？」「今日の小さな勝利は？」 | テキストプロンプト + 自由入力欄 |
| **Gratitude** | 感謝日記 | 3つの感謝項目 + 自由テキスト |
| **Health & Fitness** | 運動記録、食事記録 | 構造化フィールド + 自由テキスト |
| **Travel** | 旅行記録 | 位置情報 + 写真グリッド + テキスト |
| **Daily Log** | シンプルな日記 | 天気 + ムード + テキスト |
| **Custom Templates** | ユーザー作成 | 任意の構造を保存して再利用 |

**特筆すべき点**: テンプレートは「入力の型」を提供するだけで、すべて任意。空白のまま保存しても、後から追記しても、テンプレートを無視してもよい。この「緩さ」がほぼ日手帳の「書かなくてもいい」思想と共通する。

### 3.6 自動補完機能（Auto-completion Features）

Day One の最大の差別化要因の一つ。ユーザーが何も入力しなくても、エントリーに以下のメタデータが自動付与される：

| データ | 取得方法 | 表示形式 |
|--------|---------|---------|
| **天気** | 位置情報 + Weather API | アイコン + 気温（例: 晴れ 22度C） |
| **位置情報** | GPS | 地名 + 地図ピン |
| **歩数** | HealthKit/Google Fit 連携 | 数値（例: 8,234歩） |
| **再生中の音楽** | Apple Music/Spotify 連携 | 曲名 + アーティスト |
| **移動距離** | GPS トラッキング | km表示 |
| **日の出/日の入り** | 位置情報ベース計算 | 時刻表示 |
| **月齢** | カレンダー計算 | アイコン表示 |
| **作成デバイス** | 自動検知 | iPhone 15 Pro 等 |
| **写真のEXIF** | 写真メタデータ | 撮影日時・カメラ設定 |

**Beauty Log への最重要示唆**: 天気・湿度の自動取得は「メイクの崩れやすさ」「スキンケアの推奨」に直結する。alche:me のPRDに天気連動アドバイスが P2 で計画されているが、まず**記録時の自動取得**は P1 レベルで実装すべき。

### 3.7 ガミフィケーション/習慣形成要素

Day One は露骨なガミフィケーションは避けつつ、巧妙なリテンション設計をしている：

| 要素 | 実装方法 | 心理メカニズム |
|------|---------|---------------|
| **ストリーク（連続記録）** | 統計画面で連続記録日数を表示。ただし画面の中心ではなく、控えめな位置に配置 | ストリーク維持のモチベーション。ただし途切れても罰がない |
| **On This Day** | 「1年前の今日」「2年前の今日」のエントリーを通知で表示 | ノスタルジア効果。過去のエントリーを読む→もっと書きたくなる好循環 |
| **統計ダッシュボード** | 総エントリー数、総単語数、総写真数、最長ストリーク等を可視化 | 蓄積の可視化。「これだけ記録してきた」という達成感 |
| **年末まとめ（Year in Review）** | 年間の記録を美しいまとめとして自動生成 | 1年の振り返り + 来年も続けようという動機付け |
| **Daily Prompts（日替わりお題）** | 毎日異なる質問が表示され、それに答える形で日記を書ける | 「何を書けばいいか分からない」問題を解消 |
| **書籍化** | ジャーナルを物理的な本として注文可能 | デジタルデータの物理化＝価値の実体化 |

### 3.8 ビジュアルデザイン言語

| 要素 | Day One の実装 | 設計意図 |
|------|---------------|---------|
| **全体のトーン** | 白背景 + 控えめなグレーテキスト。写真とユーザーテキストが主役 | ジャーナルの「キャンバス」に徹する |
| **カラーパレット** | ブランドブルー（#4A90D9）をアクセントに使用。彩度は低め | 落ち着いた「書斎」のような雰囲気。刺激よりも集中 |
| **タイポグラフィ** | 本文にはセリフ体（Georgia系）を使用。日記らしい「読む」体験 | ジャーナルの格調と読みやすさの両立 |
| **写真の扱い** | エントリー内で大きく表示。複数枚はグリッドレイアウト | 写真がエントリーの「顔」になる。テキストなしでも写真だけで成立 |
| **余白** | 広めのマージンとパディング。要素間に十分な空白 | 「書き込む余裕」のある空間演出 |
| **アイコン** | SF Symbols ベースのミニマルアイコン。線が細く上品 | UIの存在感を消し、コンテンツを目立たせる |
| **アニメーション** | 遷移はゆるやかなフェード。派手なエフェクトなし | 「落ち着いた書く時間」の演出 |

### 3.9 リテンション（継続率）の仕組み

| フック | 詳細 | 有効性 |
|--------|------|--------|
| **リマインダー通知** | カスタマイズ可能な時間帯に、肯定的なトーンで通知。「今日の1ページを書きましょう」ではなく「ひとこと、残してみませんか？」 | 高 |
| **On This Day** | 過去のエントリーを通知で配信。「1年前の自分」を見ることで記録の価値を実感 | 非常に高 |
| **Daily Prompts** | 毎日異なる質問。書く内容に困らない | 高 |
| **Widget** | ホーム画面ウィジェットで直近のエントリーを表示。目に入る→思い出す→書く | 中〜高 |
| **自動メタデータ** | 何も書かなくても天気と場所が記録される。「白紙でも価値がある」感覚 | 非常に高 |
| **Apple Watch 連携** | 手首からワンタップで音声メモ → テキスト変換 | 中（対応デバイス限定） |
| **複数ジャーナル** | 「仕事」「プライベート」「旅行」等、目的別に分けられる。用途が増える＝使用頻度増加 | 高 |

### 3.10 振り返り/リフレクション機能

| 機能 | 説明 | 体験設計 |
|------|------|---------|
| **On This Day** | 過去の同じ日のエントリーを表示 | 朝の通知で「1年前のあなた」をリマインド。ノスタルジアが記録の動機に |
| **統計（Stats）** | 総エントリー数、写真数、最長ストリーク、曜日別記録頻度等 | データとしての振り返り。「数字」が蓄積の実感を生む |
| **タグフィルタリング** | タグでエントリーを横断検索。「#旅行」「#料理」等でテーマ別に振り返り | テーマ別の人生の棚卸し |
| **地図ビュー** | 地図上にピンが立つ。どこで何を感じたかを空間的に振り返り | 地理的な人生の足跡 |
| **Year in Review** | 年末に自動生成される年間まとめ。写真・統計・ハイライト | 年に1度の大きな振り返りイベント |
| **書籍化（Print）** | ジャーナルを製本してリアルな本に。自分の日記が「作品」になる | デジタルの記録を物理的な「遺産」に変換する唯一無二の体験 |

### 3.11 写真統合パターン

Day One における写真は「おまけ」ではなく「主役級」の扱い：

| パターン | 説明 |
|---------|------|
| **インライン写真** | テキスト中に写真を挿入可能。写真の前後にテキストを配置できる |
| **写真グリッド** | 複数写真は自動でグリッドレイアウト（2枚なら2列、3枚なら2+1、4枚なら2x2） |
| **写真ファーストエントリー** | テキストなし・写真だけのエントリーも「立派な記録」として表示 |
| **カメラ直接起動** | エントリー作成時にカメラを直接起動可能。撮影→保存の最短フロー |
| **既存写真の日付自動判定** | カメラロールから写真を選ぶと、EXIF日付をエントリー日として自動提案 |
| **写真からのエントリー生成** | 写真を共有→Day One→写真の日付でエントリー自動生成 |

### 3.12 モバイル固有の設計判断

| 設計判断 | 理由 |
|---------|------|
| **FAB（+ボタン）は右下** | 右手親指の自然な位置。記録開始のフリクション最小化 |
| **スワイプでエントリー間移動** | 左右スワイプで前後のエントリーに遷移。日記を「めくる」感覚 |
| **プルダウンリフレッシュ** | 上から引っ張って最新データを取得。慣れ親しんだモバイルパターン |
| **ロック画面ウィジェット** | アプリを開かずに記録開始。Live Activities 対応 |
| **Share Extension** | 他のアプリから「共有」→ Day One に直接テキスト/写真を送信 |
| **オフラインファースト** | ネットワーク接続なしでも記録可能。後で自動同期 |
| **タイピング vs 音声** | テキスト入力と音声入力（Dictation）の両方をサポート |

## 4. クロス分析：共通する成功パターン

### 4.1 両者に共通する5つの設計原則

| # | 原則 | ほぼ日手帳での表現 | Day One での表現 | 根底にある心理 |
|---|------|-------------------|------------------|---------------|
| 1 | **記録のハードルを極限まで下げる** | 書かなくてもOK。シール1枚でも「記録」 | 写真1枚+自動メタデータだけでもOK | 完璧主義の排除。「できた」の基準を最低にする |
| 2 | **蓄積を可視化する** | 本棚に並ぶ手帳の冊数 | 統計ダッシュボード、On This Day | エンダウメント効果（保有効果）。持っているものを失いたくない |
| 3 | **振り返りが新たな記録を生む** | 去年の手帳を読み返す楽しみ | On This Day 通知、Year in Review | ノスタルジア→「もっと記録しておけばよかった」→記録習慣強化 |
| 4 | **ユーザーのコンテンツが主役** | 白いページ＝キャンバス | 白背景＋ミニマルUI | UIは「額縁」に徹する。ユーザーの記録こそが価値 |
| 5 | **罰しない、褒めない、ただ待つ** | 空白ページへの罪悪感なし | ストリーク途切れのペナルティなし | 外発的動機ではなく内発的動機を育てる |

### 4.2 差異から見える設計スペクトラム

  Day One テンプレート           ほぼ日手帳 白紙ページ

         ↑ alche:me Beauty Log が狙うべきポジション

【テクノロジー ←──────────────── アナログ】

  （機械が補助する）              （人間が全てやる）

         ここ（自動補完を最大化し、ユーザー入力を最小化）

【プライベート ←──────────────── ソーシャル】

  Day One 完全暗号化              ほぼ日手帳 SNS共有文化
  （自分だけの空間）              （見せ合う楽しさ）

         デフォルトはプライベート、共有は任意選択

## 5. alche:me Beauty Log への適用提案

### 5.1 最小記録フロー（Hobonichi の「書かなくてもOK」+ Day One の「自動メタデータ」）

```
■ 提案: Beauty Log の新・最短フロー（15秒以内）

  アプリ起動
    ↓
  「今日の顔」ボタン（FAB、常時表示）
    ↓
  セルフィー撮影 or カメラロールから選択
    ↓
  自動取得: 天気 ☀️ 26度 / 湿度 65% / 位置情報
    ↓
  5段階評価（星 or 顔アイコン1タップ）
    ↓
  保存完了
    ↓
  （任意追記: レシピリンク、使用アイテム、ムード、メモ）
```

**ポイント**: 写真+評価だけで「記録完了」とする。追加情報は後からいつでも追記可能。

### 5.2 テンプレートシステム（Day One の「Daily Prompts」のビューティー版）

| テンプレート名 | プロンプト例 | 含まれるフィールド |
|--------------|-------------|------------------|
| **Quick Snap** | 「今日の顔を1枚」 | 写真 + 評価（最小構成） |
| **Morning Routine** | 「今朝のメイクを記録」 | 写真 + レシピ + 使用アイテム + ムード |
| **Skin Check** | 「今日のお肌の調子は？」 | 肌状態チェック + スキンケアメモ |
| **Event Look** | 「特別な日のルック」 | 写真(複数) + イベント名 + 詳細メモ |
| **Discovery** | 「今日の新しい発見」 | 新しい組み合わせ/テクニック + 写真 |

### 5.3 自動補完の美容版（Day One メタデータの応用）

## 6. 現状の Beauty Log との Gap 分析

現在の alche:me Beauty Log の実装を分析し、改善すべきポイントを特定する。

### 6.1 現状の構成

現在の Beauty Log は以下の要素で構成されている：

- **カレンダービュー**: 月間カレンダーにドット表示 + カードリスト（`beauty-log-calendar.tsx`）
- **タイムラインビュー**: 写真付きカード一覧（`page.tsx` 内 `TimelineCard`）
- **入力フォーム**: 日付、レシピリンク、使用アイテム、評価、気分、天気、シーン、メモ、写真（`beauty-log-form.tsx`）
- **データ型**: `BeautyLogEntry`（日付、レシピ、使用アイテム、評価、気分、天気、シーン、メモ、写真、自動タグ）

### 6.2 Gap 一覧

| # | カテゴリ | 現状 | あるべき姿 | ギャップの深刻度 |
|---|---------|------|-----------|----------------|
| 1 | **最短記録フロー** | フォームを開く→全フィールドが一度に表示 | 写真+評価の2ステップで完了 | **Critical** — 日次記録の最大のハードル |
| 2 | **自動メタデータ** | 天気は手動選択（晴れ/曇り/雨/雪の4択） | Weather API で自動取得（気温・湿度含む） | **High** — 入力負担の削減に直結 |
| 3 | **振り返り機能** | なし（カレンダーとタイムラインの閲覧のみ） | 週次/月次サマリー、On This Day | **High** — 継続率のエンジンが欠如 |
| 4 | **FAB（常時記録ボタン）** | ヘッダー内の「記録」ボタン | FAB として常時表示 | **Medium** — アクセス性の改善 |
| 5 | **写真ファースト設計** | 写真はフォームの下部に配置 | 写真が記録フローの最初のステップ | **High** — ビューティーアプリとして写真が主役であるべき |
| 6 | **テンプレート** | なし | Quick Snap / Morning Routine 等 | **Medium** — 記入のハードル低下に寄与 |
| 7 | **ガミフィケーション** | なし | ストリーク、パレット、バッジ | **Medium** — 継続の動機付け |
| 8 | **日替わりコンテンツ** | なし | Beauty Tip / Color of the Day | **Low** — アプリを開く動機の補強 |
| 9 | **通知/リマインダー** | なし | 朝/夜の記録リマインド | **High** — 習慣形成の基本 |
| 10 | **スワイプナビゲーション** | なし（日付クリックで遷移） | 日別ログ間のスワイプ移動 | **Low** — UX の洗練 |

| 12 | **検索/フィルタリング** | なし | タグ、気分、天気、評価等での絞り込み | **Medium** — 蓄積データの活用 |

## 補足: 主要な設計判断ポイント

### Q1: カレンダービューとタイムラインビュー、どちらをデフォルトにするか？

**推奨: カレンダービューをデフォルトに維持**

- ほぼ日手帳の成功はカレンダー（1日1ページ）の「コレクション感」に支えられている
- カレンダーは「空白の日」が見え、「埋めたい」という心理が働く
- ただし Day One のようにタイムラインへの切替は1タップで（現状の実装で良い）

### Q2: 評価（星）は5段階のままでいいか？

**推奨: 5段階を維持しつつ、顔アイコンも検討**

- 5段階は十分な粒度がある
- ただし「星」はレストランレビューの連想が強い。Beauty Log では「顔アイコン」（とても良い/良い/普通/いまいち/最悪）の方が直感的かもしれない
- A/Bテストでの検証を推奨

### Q3: 「書かない日」をどう扱うか？

**推奨: ほぼ日手帳の「書かなくてもOK」思想を採用**

- 空白の日に「未記録」等のネガティブな表示をしない
- カレンダー上で記録日はカラフルに、未記録日はニュートラルな白/グレーで表示
- ストリークが途切れても「最長記録」は保持され、次のストリーク開始を肯定的に表示

### Q4: ビジュアルデザインのトーンは？

**推奨: 現在のalche:meデザインシステムを維持しつつ、ジャーナル的な要素を追加**

- alche:me の rose/gold/charcoal のカラーパレットはビューティーアプリとして適切
- Day One の「コンテンツが主役」思想を取り入れ、UIクロームを最小化
- 写真が大きく表示される設計に（現在のTimelineCardの写真サイズをさらに拡大）
- タイポグラフィに少しセリフ体の要素を加え、「日記帳」の温かみを演出
